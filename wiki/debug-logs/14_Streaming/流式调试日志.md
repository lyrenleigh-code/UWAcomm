## 2026-04-15 · P1 实施期

### 坑 1：MATLAB R2025b 链式赋值陷阱

**现象**：`p1_demo_ui` 启动报"函数或变量 'uilabel' 无法识别"，但 `which uilabel` 显示路径正常存在。

**根因**：MATLAB 解析函数体时静态分析每个标识符。`uilabel(...).Layout.Row = X` 形式让 MATLAB 把 `uilabel` 误判为变量（出现在赋值左侧位置），整函数所有 `uilabel(...)` 调用都被视作"未赋值变量索引"。

**修复**：永远不要链式赋值函数返回值的属性 — 用中间变量：
```matlab
% 错：
uilabel(grid, 'Text', 'X').Layout.Row = 1;
% 对：
lbl = uilabel(grid, 'Text', 'X');
lbl.Layout.Row = 1;
```

### 坑 2：uigridlayout 不接受 CSS 语法

**现象**：`mid.ColumnWidth = {'1fr', '1fr'}` 报错 `set.ColumnWidth (line 155)`。

**根因**：MATLAB uigridlayout 用自己的 unit 系统：`'1x'`/`'2x'`（比例权重）/ 数字（像素）/ `'fit'`（自适应）。`'1fr'` 是 CSS Grid 语法，不被识别。

**修复**：`{'1x', '1x'}`。

### 坑 3：cell 数组与数值矩阵混淆

**现象**：visualize_p1_frame 的 segs 数组定义后 `segs{s, 3}` 报错 `Cell contents reference from a non-cell array object`。

**根因**：用 `[ ]` 拼装混合数字+字符串：
```matlab
segs = [0, fr.N_pre, 'HFM+'; ...]   % 'HFM+' 被转成 ASCII [72 70 77 43]
```
变成纯数值矩阵，不能用 `{}` 索引。

**修复**：用 cell 语法 `{ }`：
```matlab
segs = {0, fr.N_pre, 'HFM+'; ...}
```

## 2026-04-15 · P2 实施期

### 坑 4：downconvert LPF 暖机吃首帧

**现象**：P2 多帧测试中 frame 1 经常 CRC 失败，frame 2+ 正常。

**根因**：`downconvert` 内部 LPF 是 64 阶 FIR，前 ~64 样本是滤波瞬态，会损坏 frame 1 的 HFM+ 信号（位于样本 1..2400）。

**修复**：rx_stream_p2 在 downconvert 前预填零给 LPF 暖机：
```matlab
N_lpf_warmup = 200;
rx_pb_padded = [zeros(1, N_lpf_warmup), rx_pb(:).'];
[bb_padded, ~] = downconvert(rx_pb_padded, sys.fs, sys.fc, sys.fhmfsk.total_bw);
bb_raw = bb_padded(N_lpf_warmup+1 : end);  % 跳过暖机区
```

### 坑 5：阈值化检测对 Jakes 衰落首帧不鲁棒

**现象**：Jakes slow 衰落下 frame 1 漏检（Stage A.5 不打印强制锚定消息），text 输出 `[missing frame 1]信测试...`。

**根因**：自适应阈值 = `max(K·noise, ratio·peak_max)`。`peak_max` 来自最强的中间帧。frame 1 在 Jakes 深衰落下峰值可能只有 peak_max 的 10–20%，被全局阈值过滤。

**修复**：frame_detector V1.1 改 hybrid 模式 — frame 1 直接在预期窗口 `[1, 30%·frame_len + N_template]` 取**绝对最大**作为锚点（不依赖阈值），保证 frame 1 一定被尝试解码。

### 坑 6：硬判决 FSK 在浅衰落下脆弱

**现象**：frame 1 检测成功（k=1），但 header CRC 失败（crc=0 magic=1）。

**根因**：modem_decode_fhmfsk V1.0 用硬判决 `(2*bit-1)*10` 给 Viterbi。Jakes 浅衰落下 1 位错就 CRC 挂，没有置信度信息辅助 Viterbi 纠错。

**修复**：升级 V1.1 用每符号 8 频率能量直接算 per-bit LLR：
```matlab
LLR(b_j) = (max_e_b1 - max_e_b0) / median(e_freqs)
```
配合 [7,5] 卷积码 dfree=5 能纠多位错。

### 坑 7：FH-MFSK 无均衡 → 大延时多径必崩

**现象**：原"5径 深衰减"preset（延时展宽 1.5ms = 75% sym）下整段失败。

**根因**：FH-MFSK 用 FFT 能量检测，每符号 2ms 窗口。延时展宽 > 50% 符号时长就严重 ISI，前一符号的多径拖尾打散到当前符号 FFT 的多个频率，软/硬判决都救不回。

**修复**（demo 妥协）：调整 preset 把延时展宽缩到 0.6ms（30% sym），平衡"挑战性"与"可解码性"。

**根本解决**：等 P3 接入 OFDM/SC-FDE，那些体制带正经的均衡器，能扛 1.5ms+ 延时展宽。

### 坑 8：gen_doppler_channel V1.0 基带相位公式 fs/fc 倍率错

**现象**：
P4 接入 `gen_doppler_channel` 后（commit 前版本 V1.0），UI 中 `doppler=12 Hz + tv_enable=false + SC-FDE + 6径标准 + SNR=15` 下 BER 接近 50%，而 P3 同参数历史基线是 **0.1%**（[[../../comparisons/e2e-timevarying-baseline|4-19 A2 基线]]、[[../../../specs/active/2026-04-20-alpha-compensation-pipeline-debug.md|4-20 pipeline debug]] L19-24）。

**定位推理**：
- P3 路径：`conv(h_tap) → comp_resample_spline(α) → exp(j·2π·dop_hz·t)`，其中 `dop_hz = α·fc`（12 Hz）。
- P4 V1.0 路径（`gen_doppler_channel.m:95`）：`phase_shift = 2π·α·fs·t_stretched`，等价于在基带施加 `α·fs ≈ 48 Hz` 的旋转（fs=48k fs/fc=4× 过快）。
- P4 传入 `dop_hz=12 Hz` 实际被 RX 感受为 `dop=48 Hz`，已超过 4-19 / 4-20 记录的 **24 Hz 物理断崖**（α=2e-3 对应 dop=24 Hz，BER 47%）。

**V1.1 修复**（`gen_doppler_channel.m`）：
- 新增第 7 参 `fc`（可选，向后兼容）
- 传入后公式改为物理正确：`phase_shift = 2π·fc·cumsum(α_t)/fs`
- 同步修正 `t_stretched` 起点从 0 开始（原 V1.0 起点 1/fs，导致 α=0 时也有 1 样本偏移）
- `snr_db=Inf` 显式跳过内部加噪（便于外层 FIFO 层独立加噪）
- 未传 `fc` 时 `warning('gen_doppler_channel:NoFc', ...)` 并退回 V1.0 行为（保护 test_doppler / gen_uwa_channel_array 等遗留 caller）
- `p4_demo_ui.m` on_transmit 改 `gen_doppler_channel(..., app.sys.fc)`

**单元验证**（`test_p4_channel_smoke.m` case 6）：
- DC 基带 s=ones(1024) + paths=单径 + α=1e-3
- 期望：V1.1 相位频率 = fc·α = 12 Hz，V1.0 = fs·α = 48 Hz
- 单元层面两个公式都能独立复现正确值 ✅

### 坑 8-延伸：多径叠加与 resample 顺序错 → per-path 时延偏差

**现象**（`diag_p3_vs_p4_frame_results.txt` 2026-04-22 20:10）：

| preset | α=0 | α=5e-4 | α=1e-3 | α=2e-3 |
|:-------|:---:|:------:|:------:|:------:|
| AWGN（单径） | 4e-10 | 0.096 | 0.095 | 0.095 |
| 6径标准 | 4e-10 | **0.176** | **0.284** | **0.431** |

**定位**：V1.1 channel 物理正确（`diag_p4_doppler_isolate` 全 PASS），但样本对比失败。原因是**多径叠加和 Doppler resample 的顺序错**：

- **P3 顺序**（对）：`conv(h_tap) → resample → phase`
  - 多径延时在原 fs 时间轴累加；Doppler 对整体接收信号伸缩
  - 读位置：`n/(1+α) − τ_p`
- **P4 V1.1 顺序**（错）：`resample(s) → phase → per-path 延时叠加`
  - 多径延时在 resample 后的时间轴上施加（每路径不同的偏差）
  - 读位置：`(n−1)/(1+α) − τ_p/(1+α)`
- **逐路径差** = `1/(1+α) − α·τ_p/(1+α)`：
  - path 0 约 1 样本偏移
  - 最远 path（τ_p=720，α=1e-3）约 0.28 样本
  - **每条 path 偏移不同 → RX h_est 每 tap 错位 sub-sample → Turbo 发散**

**物理依据**：多径是信道自身的 LTI 属性；Doppler 是平台运动对整体接收信号的时间缩放。`conv` 应在 resample 之前。

**V1.2 修复**（`p4_demo_ui.m` on_transmit）：
```matlab
% Step 1: 多径 conv（原 fs 时间轴）
frame_mp = conv(frame_bb, h_tap);
frame_mp = frame_mp(1:length(frame_bb));
% Step 2: 单径喂 gen_doppler_channel，只让它做 α(t) 伸缩 + fc 相位
paths_single = struct('delays', 0, 'gains', 1);
[frame_ch_raw, ch_info] = gen_doppler_channel( ...
    frame_mp, app.sys.fs, alpha_b, paths_single, Inf, tv, app.sys.fc);
```

同步更新 `diag_p3_vs_p4_frame.m` 的 P4 路径为 conv→gen_doppler，**再跑应当 8/8 PASS**（α=0 接近精度、α>0 仅残留 sub-sample 数值误差 < 0.01）。

**注**：`gen_doppler_channel` 的 paths 多路径 loop 保持不变（其他 caller 如 test_doppler / gen_uwa_channel_array 还在用），只是 P4 集成层不再依赖它做多径。

### 坑 8-再延伸：RX 原本就没 α 反补偿（P3/P4 共性限制）

**现象**（2026-04-22 20:27 用户测试）：
- P4 dop=+6Hz SC-FDE 6径 SNR=15 → **BER=50.277%**
- **P3 同参数 → BER=49.899%**（同等垃圾）

**定位**：
- `try_decode_frame()` 调 `detect_frame_stream`，**读出 `sync_det.alpha_est` 却不用**
- `modem_decode_scfde` 内部无 α 估计/补偿
- 4-19 / 4-20 基线 "α=5e-4 → 0% BER" 来自 **`test_scfde_discrete_doppler.m` / `test_scfde_timevarying.m`**，那两个脚本**显式调 `est_alpha_dual_chirp + comp_resample_spline`**。UI 从来没调。
- P3 老 on_transmit 有注释：`% RX 若无 α 反补偿 + 时变信道估计 → 高 Doppler BER 崩溃（已知限制）`

**P4.1 修复**（`p4_demo_ui.m` try_decode_frame）：
```matlab
alpha_est_rx = sync_det.alpha_est;   % 已由 detect_frame_stream 计算
if abs(alpha_est_rx) > 1e-6 && alpha_conf > 0.3
    rx_seg = comp_resample_spline(rx_seg, -alpha_est_rx, app.sys.fs, 'fast');
    rx_seg = rx_seg(1:fn);           % 截回 fn 长度保持下游兼容
end
```

符号约定：detect_frame_stream.alpha_est 与 comp_resample_spline 一致（"正=压缩"），补偿传 `-α_est` 做反向伸缩。

**注**：只修 P4，**不动 P3**（P3 是稳定参考 demo）。
