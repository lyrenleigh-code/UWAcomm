# 14 流式仿真框架

> 返回 [[index]] | [[uwacomm-moc]]
> 代码 README: `modules/14_Streaming/README.md`
> Master spec: `specs/active/2026-04-15-streaming-framework-master.md`

## 目的

模拟真实水声通信部署：文本输入 → wav 文件传输 → 信道模拟 → wav 文件接收 → 文本输出。

## 与批处理测试（13_SourceCode）的区别

| 维度 | 13_SourceCode/tests | 14_Streaming |
|------|---------------------|--------------|
| 输入 | 随机 bits（数组） | 用户文本（UTF-8） |
| 传输介质 | in-memory 数组 | wav 文件 |
| 信道 | 同进程调用 gen_uwa_channel | 独立 channel daemon 进程 |
| 帧定位 | 已知偏移 | 滑动匹配滤波发现 |
| 多帧 | 否 | 是（长文本分帧） |
| 体制切换 | 手动配置 | AMC 自适应 |
| 并发 | 串行脚本 | 三进程并发 |
| 自组网 | 无概念 | 帧头留节点地址字段 |

## 核心概念

### 会话目录（方案 B）

每次通信任务创建 `session_<timestamp>/`，三进程通过该目录协作：
- TX 写 `raw_frames/NNNN.wav` + `NNNN.ready`（close 后原子创建）
- Channel daemon 读 raw_frames，写 `channel_frames/NNNN.wav` + `NNNN.ready`
- RX 读 channel_frames，写 `rx_out/NNNN.meta.json` + `session_text.log`

每帧独立 wav 文件避免 Windows 文件锁冲突。

### 帧协议

物理帧 = [HFM+|guard|HFM-|guard|LFM1|guard|LFM2|guard|Header|Payload]

- Header (16B) 用 FH-MFSK 调制，最鲁棒，保证控制信息永远可解
- Payload 按 Header.scheme 字段指定的体制调制
- 每段独立 CRC16 校验

### 统一 modem API

6 体制包装为 `modem_encode(bits, scheme, sys)` / `modem_decode(rx_pb, scheme, sys, meta)`，通过 `scheme` 字段分发。

### AMC（物理层指标）

RX 估计链路质量（LFM sync_peak / SNR / delay_spread / doppler）→ 决策表 → 推荐体制 → 通过 ACK 反馈 TX → TX 切换下一帧 scheme。
